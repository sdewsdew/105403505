一、背景

Structured Query Language(SQL) 於1980年代成為American National Standard Institute (ANSI)和International Organization for Standardization (ISO)的標準，為用於管理、操作關聯式資料庫的語言，包括資料庫的新增、刪除、更新和查詢等動作。SQL應用於許多應用程式中(例如：使用者登入、購物網站等)，做為和Database Management System (DBMS)溝通的方式，如圖1所示。

圖1 應用程式與DBMS間的關係

 

在以上圖1的架構下，不曾聽聞SQL Injection的發生，因為資料庫所有的新增、刪除、更新和查詢都在database application程式設計師的控制之下，不會自找麻煩，製造問題給自己。

 

1.         SQL程式演變與SQL Injection環境

但自網際網路盛行起，圖1的database application分為前端和後端兩部份，如圖2所示。伺服器接受使用者的輸入，並使用此輸入去查詢DBMS，因此這包含了3個元素：Client、Server和DBMS。從此，database application送到DBMS的指令就不完全由程式設計師所掌握。

圖2以一個網站上的使用者登入為例，程式設計師在網頁伺服器的後端(例如：PHP、ASP、Java和Python等)撰寫的應用程式內預先寫了一個查詢使用者資料的SQL語句，不過必須等到伺服器收到使用者輸入的名稱、密碼後，才能完成這一個查詢語句(因為這時才知道要查詢誰的資料)，接著再將此語句送給DBMS執行，伺服器收到DBMS回傳的結果，對此進行處理後再呈現給使用者(登入成功或失敗)。此為正常情況下使用者的登入過程。

2.         SQL Injection問題所在

會發生SQL Injection的問題在於後端程式使用由client提供的外部資料動態組成SQL語句去查詢DBMS，且未對這些資料進行檢查。因為這些資料可由使用者自行輸入，攻擊者能在裡面輸入與SQL有關的指令、或是用相關語法去改變程式開發者原先寫好的SQL語句結構，而這些指令或是結構被更改的語句就會被送進DBMS內執行，產生與database application程式設計師預期不同的結果。

圖3繼續使用與圖2相同的使用者登入來描述SQL Injection的發生。假設攻擊者不知道有哪些使用者名稱、密碼可以登入，他可以藉由輸入與SQL相關的語法，改變SQL語句的邏輯。程式設計師原本預期的SQL查詢行為是「回傳名字、密碼與使用者的輸入相符的那一筆資料(record)」，而攻擊者輸入「a’ or 1=1#」後就可以改變語句的邏輯，將SQL查詢行為變成「不管名字與使用者的輸入是否相符，全部都回傳」(細節會在下一個主題「攻擊種類」中做說明)。如此一來DBMS會回傳table內所有的資料給後端程式，而因為程式判斷帳密是否存在的方式就是看DBMS是否有回傳東西，因此這時即使回傳的資料與查詢的資料不同，也會認為使用者輸入的帳密是存在的，進而讓他成功登入。

3.         SQL Injection重要性

SQL Injection是網頁中常見的攻擊方式，根據Akamai於2017年11月至2019年3月間至他的用戶中收集的資料統計，網頁遭受的攻擊有65.1%來自SQL Injection。OWASP (Open Web Application Security Project)為一個致力於網頁安全的非營利組織，自2003年起會不定期的發佈其評比的前10大(OWASP Top10)網頁威脅。而SQL Injection(列為Injection攻擊)自第一版到目前最新版的2017 Top10中皆在榜上，詳見表1。

SQL Injection 可未授權存取、竊取和竄改資料，若權限足夠也能在網站中插入惡意連結與代碼，或達到破壞、癱瘓主機等目的。攻擊者甚至可透過DBMS執行與作業系統相關的指令。以下我們將分成union-based、blind injection、OS interaction等主題，以程式實例說明SQL Injection的原理。
